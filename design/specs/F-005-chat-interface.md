# F-005: Chat Interface

**Spec Version:** 1.0
**Date:** 2026-03-16
**ADR Reference:** ADR-005, ADR-006

## Overview

A real-time chat interface that allows users to query their knowledge base using natural language. Responses are generated by the LLM using hybrid retrieval (vector + graph) as context, with full source attribution on every answer. Supports streaming responses via WebSocket.

## User Stories

- As a user, I want to ask natural language questions about my knowledge base and get accurate answers
- As a user, I want to see exactly which documents and sources informed each answer
- As a user, I want responses to stream in real-time, not wait for the full response
- As a user, I want my conversation history preserved within a session
- As a user, I want to start new conversations without losing old ones

## Functional Requirements

### FR-1: WebSocket Chat Endpoint

```python
# Backend WebSocket endpoint
@app.websocket("/ws/chat/{session_id}")
async def chat_websocket(websocket: WebSocket, session_id: str):
    """
    Message protocol:
    Client → Server: {"type": "query", "content": "What did we decide about the API?"}
    Server → Client: {"type": "chunk", "content": "Based on..."} (streaming)
    Server → Client: {"type": "sources", "sources": [...]} (after generation)
    Server → Client: {"type": "done"}
    Server → Client: {"type": "error", "message": "..."}
    """
```

### FR-2: Query Processing Pipeline

1. Receive user query via WebSocket
2. Perform hybrid retrieval (F-002):
   - Vector search in LanceDB for semantically similar chunks
   - Entity extraction from query → graph traversal in Kuzu
3. Assemble context from retrieval results
4. Construct LLM prompt with system instructions, context, and user query
5. Stream LLM response back via WebSocket
6. After generation: send source attribution data

### FR-3: Source Attribution

Every response must include citations:

```python
class SourceAttribution(BaseModel):
    source_type: SourceType
    source_id: str
    title: str
    excerpt: str              # Relevant excerpt from the source
    relevance_score: float
    timestamp: datetime
    url: str | None           # Deep link to original source
    participants: list[str]   # For meetings
```

- Citations are numbered in the response text: "The API design was finalized in last week's meeting [1]"
- Source cards appear below the response with full details

### FR-4: Conversation Management

```python
class ChatMessage(BaseModel):
    id: str
    role: str                 # "user" or "assistant"
    content: str
    sources: list[SourceAttribution] = []
    timestamp: datetime

class ChatSession(BaseModel):
    id: str
    title: str                # Auto-generated from first message
    messages: list[ChatMessage]
    created_at: datetime
    updated_at: datetime
```

- Sessions are persisted to disk (SQLite or JSON)
- Session list is available via API
- Old sessions can be resumed
- Session title auto-generated from the first user query

### FR-5: Follow-up Suggestions

After each response, suggest 2-3 relevant follow-up questions based on:
- Entities mentioned in the response
- Related topics in the knowledge graph
- Common query patterns

### FR-6: Query Modes

| Mode | Trigger | Description |
|---|---|---|
| Natural language | Default | Standard RAG: retrieve + generate |
| Graph query | `/graph ...` prefix | Direct Cypher query against Kuzu |
| Direct search | `/search ...` prefix | Vector search without LLM generation |

## Non-Functional Requirements

- First token latency < 2 seconds after query submission
- Streaming must not buffer — each token sent as generated
- Chat UI must be responsive on desktop browsers (1024px+)
- Conversation history must survive page refresh
- WebSocket must auto-reconnect on disconnection

## UI/UX Requirements

### Layout
- Full-width chat panel with sidebar for session list
- Message bubbles: user messages right-aligned, assistant messages left-aligned
- Source cards below assistant messages (collapsible)
- Typing indicator during LLM generation
- Follow-up suggestion chips below the response

### Message Rendering
- Markdown rendering in assistant messages (code blocks, lists, tables, bold/italic)
- Syntax highlighting in code blocks
- Inline citation numbers `[1]` clickable to highlight the corresponding source card
- Copy button on code blocks and full responses

### Input Area
- Multi-line text input with Shift+Enter for new lines, Enter to send
- Send button
- Query mode indicator (natural language / graph / search)

## API Endpoints

| Method | Path | Description |
|---|---|---|
| WebSocket | `/ws/chat/{session_id}` | Real-time chat |
| GET | `/api/chat/sessions` | List all sessions |
| GET | `/api/chat/sessions/{id}` | Get session with messages |
| POST | `/api/chat/sessions` | Create new session |
| DELETE | `/api/chat/sessions/{id}` | Delete a session |

## Testing Strategy

### Unit Tests
- Test query processing pipeline with mocked retrieval results
- Test source attribution assembly
- Test conversation session management (create, list, delete)
- Test follow-up suggestion generation
- Test query mode detection (natural language vs. /graph vs. /search)
- Test LLM prompt construction with context

### Integration Tests
- Test WebSocket connection lifecycle (connect, message, disconnect)
- Test streaming response delivery
- Test session persistence across application restarts
- Test hybrid retrieval integration in chat context

### Frontend Tests
- Test message rendering (markdown, citations, code blocks)
- Test WebSocket reconnection logic
- Test session switching
- Test input handling (Enter vs Shift+Enter)
