# How It Works

This page explains what happens under the hood when you sync a connector, why chunking and embeddings matter, what the graph database stores, what the LLM is used for, and where SQLite fits in.

---

## The big picture

Nexus PKM has two main pipelines:

1. **Ingestion** — takes raw documents from your sources and builds the knowledge base
2. **Retrieval** — takes your questions and searches the knowledge base to produce answers

```
Source (Teams / Outlook / Obsidian / JIRA / Apple Notes)
  │
  ▼ Connector fetches and normalises
  │
  ▼ Chunking
  │
  ├──► Embedding ──► LanceDB (vector store)
  │
  └──► Entity extraction (LLM) ──► Kuzu (graph database)


User query
  │
  ├──► Vector search (LanceDB) ──► semantically similar text chunks
  │
  └──► Graph traversal (Kuzu) ──► connected entities and documents
          │
          ▼
        Result merging + ranking
          │
          ▼
        LLM generates answer with source citations
```

---

## Step 1 — Connectors: getting data in

Each connector fetches content from one source system and converts it to a common document format. All connectors are:

- **Read-only** — they never write back to the source
- **Incremental** — after the first sync they only pull new or changed content
- **Scheduled** — they run automatically in the background on a configurable interval

See the [Connectors](connectors.md) guide for per-connector setup and details.

---

## Step 2 — Chunking: splitting documents into pieces

A meeting transcript or email thread can be thousands of words. Embedding and retrieving an entire document as a unit doesn't work well — the signal gets diluted and long documents exceed LLM context limits.

Nexus PKM splits each document into **overlapping chunks** before indexing:

| Setting | Default | Location |
|---|---|---|
| Chunk size | 512 tokens | `config/app.yaml` → `chunking.size` |
| Overlap | 50 tokens | `config/app.yaml` → `chunking.overlap` |

The overlap ensures that sentences that fall near a chunk boundary still appear together in at least one chunk, preserving context.

---

## Step 3 — Embeddings: turning text into vectors

Once a document is split into chunks, each chunk is converted to a **vector embedding** — a list of numbers that encodes the semantic meaning of the text. Two chunks about the same topic will have similar vectors even if they use different words.

Embeddings are generated by the **embedding model** configured in `config/providers.yaml`. The default is AWS Bedrock Titan (1024 dimensions). See [LLM Providers](llm-providers.md) for other options.

The vectors are stored in **LanceDB**, a local file-based vector database at `data/lancedb/`. This is what powers semantic search.

> **Important:** the embedding dimensions are fixed at index time. If you switch to an embedding model with different dimensions, you must delete `data/lancedb/` and re-ingest. See the LLM provider guide for details.

---

## Step 4 — Entity extraction: building the knowledge graph

After a document is embedded, it enters an **asynchronous extraction queue**. The LLM reads each document chunk and extracts structured information:

**Entity types extracted:**

| Type | Examples |
|---|---|
| Person | "Jane Smith", "jsmith@company.com", "@jane" |
| Project | "Project Atlas", "NXP", dedicated Slack channels |
| Topic | Recurring themes, meeting agenda items, hashtags |
| Decision | "We decided to...", "The agreed approach is..." |
| ActionItem | "TODO: ", "John will...", JIRA tickets |
| Meeting | Calendar events, transcript headers |

**Relationships extracted:**

- Who attended which meeting
- Who is assigned to which action item
- Which decisions were made in which meeting
- Which documents are about the same topic
- Who works on which project
- How action items block or follow up on each other

All of this is stored in **Kuzu**, a local graph database at `data/kuzu/`. Kuzu supports Cypher queries, the same query language used by Neo4j.

### Deduplication

If "John Smith" appears in a meeting transcript and "John" appears in an email, the system resolves them to the same Person node using email matching, exact name matching, fuzzy matching, and — for ambiguous cases — an LLM-assisted comparison.

### Contradiction detection

When new information conflicts with existing knowledge (e.g. a deadline changes, or ownership of a project changes), the system flags it as a contradiction and surfaces a notification so you can review it.

---

## Step 5 — Hybrid retrieval: answering queries

When you ask a question (via chat or search), Nexus PKM runs two searches simultaneously and combines the results:

### Vector search (LanceDB)

Your query is embedded using the same embedding model used at ingest time. LanceDB finds the most semantically similar chunks using cosine similarity. This catches relevant content even when it uses different words than your query.

### Graph traversal (Kuzu)

The system extracts entities from your query, then traverses the knowledge graph to find connected entities and documents within two hops. This surfaces
content that's related by relationship rather than text similarity — for example, all documents involving the same person or project.

### Ranking

Results from both sources are merged and ranked using a weighted score:

```
combined_score = (vector_score × 0.6) + (graph_score × 0.3) + (recency_score × 0.1)
```

The weights are configurable in `config/app.yaml` under `retrieval`.

---

## The LLM's roles

The LLM is used in three distinct ways:

| Role | When | What it does |
|---|---|---|
| **Entity extraction** | After each document is ingested | Reads document chunks and outputs structured JSON of entities and relationships |
| **Query answering** | When you use chat | Reads the retrieved context and generates a natural language answer with citations |
| **Deduplication** | When entity matching is ambiguous | Determines whether two entity references refer to the same thing |

All three use the same provider configured in `config/providers.yaml`. A more capable model produces better entity extraction and reasoning. A faster local model is good enough for most conversational queries.

---

## What SQLite stores

SQLite is used for lightweight structured data that doesn't need vector or graph capabilities:

| Data | Location |
|---|---|
| Chat session history | Managed by the backend |
| Notifications | Contradiction alerts, meeting prep, related content |
| Connector sync state | Last sync timestamps, delta tokens |

Unlike LanceDB and Kuzu, the SQLite database is not stored in a fixed path you configure — it's managed internally by the application.

---

## Storage layout

```
data/
├── lancedb/          # Vector store — chunk embeddings
├── kuzu/             # Graph store — entities and relationships
└── .tokens/          # Encrypted OAuth2 tokens (Teams, Outlook)
                      # Gitignored — never committed
```

The `data/` directory is gitignored. Back it up to preserve your knowledge base.

---

## Re-indexing

If you change your embedding model, or want to rebuild the knowledge base from scratch:

```bash
# If you've switched embedding models, delete the vector store first:
rm -rf data/lancedb/
# Then restart the backend — the next sync re-embeds everything

# Full re-index from stored raw documents (no re-fetch from sources)
curl -X POST http://127.0.0.1:8000/api/engine/reindex
```

The graph database (`data/kuzu/`) is unaffected by re-indexing unless you explicitly delete it.
